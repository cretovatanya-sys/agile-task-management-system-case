# Внедрение системы управления задачами на основе Agile для команды внедрения

## Обзор проекта

Данный проект описывает внедрение структурированного процесса управления задачами для команды внедрения, работающей над развертыванием и настройкой медицинской информационной системы.

В рамках проекта команда внедрения занималась настройкой системы в соответствии с бизнес-процессами заказчика, а также взаимодействовала с командой разработки по вопросам доработок и исправлений системы.

Изначально управление внутренними задачами команды было неструктурированным: задачи передавались через сообщения в Telegram-чате. Такой подход создавал сложности с отслеживанием задач, их приоритизацией и координацией работы команды.

Для повышения эффективности и прозрачности рабочего процесса мной была внедрена структурированная система управления задачами на основе Agile-подхода, включающая ведение backlog, планирование недельных спринтов и использование Kanban-доски.

---

## Бизнес-проблема

До внедрения системы управления задачами команда сталкивалась со следующими проблемами:

- задачи передавались через сообщения в Telegram
- отсутствовало централизованное хранилище задач
- отсутствовала система приоритизации
- задачи могли теряться в истории сообщений
- не была определена ответственность за выполнение задач
- было сложно отслеживать статус выполнения задач
- отсутствовала прозрачность загрузки команды
- взаимодействие с командой разработки было неструктурированным

Все это снижало прозрачность процессов и усложняло управление задачами в рамках проекта внедрения.

Старший специалист группы внедрения / Team Lead

Мои основные задачи:

- взаимодействие с заказчиком для уточнения бизнес-процессов
- декомпозиция задач по настройке системы
- формирование и ведение backlog задач
- приоритизация задач команды
- организация недельных спринтов
- распределение задач между инженерами внедрения

## Анализ текущего процесса (AS IS)

Изначально управление задачами в команде происходило неформально и строилось на переписке в мессенджере.

Типовой процесс выглядел следующим образом:

1. Заказчик формулирует запрос или проблему
2. Информация передается в Telegram-чат
3. Задачи обсуждаются в переписке
4. Члены команды самостоятельно выбирают задачи из сообщений
5. Статус выполнения задач нигде централизованно не фиксируется

Основные проблемы процесса:

- отсутствие структурированного учета задач
- отсутствие приоритизации
- низкая прозрачность статусов задач
- риск потери задач в истории сообщений
- сложность координации между членами команды
- отсутствие контроля распределения нагрузки

## Реализованное решение (TO BE)

Для повышения прозрачности процессов и улучшения управления задачами мной была внедрена система управления задачами на основе Agile-подхода.

Основные элементы решения:

- централизованное ведение backlog задач
- приоритизация задач
- планирование задач в рамках недельных спринтов
- использование Kanban-доски для отслеживания статусов задач
- структурированные статусы задач
- закрепление ответственности за выполнение задач

Это позволило создать прозрачный и управляемый процесс постановки и выполнения задач внутри команды внедрения.

## Workflow управления задачами

В рамках решения была внедрена Kanban-доска со следующими статусами задач:

- Backlog  
- Ready for Sprint  
- In Progress  
- Waiting for Developer  
- Testing  
- Done  

Отдельная колонка Waiting for Developer использовалась для задач, требующих участия команды разработки, например:

- консультации разработчиков
- исправления ошибок
- доработки функциональности
- изменения логики системы

После решения проблемы разработчиками задача возвращалась в рабочий процесс команды внедрения.

## Структура backlog

Задачи в backlog структурировались с использованием следующих атрибутов:

| Задача | Приоритет | Тип | Ответственный | Статус |
|------|------|------|------|------|
| Настройка модуля | Высокий | Внедрение | Инженер | В работе |
| Запрос на интеграцию | Высокий | Разработка | Разработчик | Ожидание разработки |
| Корректировка процесса | Средний | Улучшение | Инженер | Backlog |

## Приоритизация задач

Для эффективного управления backlog задачи приоритизировались на основе их бизнес-ценности и срочности.

Использовались следующие уровни приоритета:

- Высокий - критично для работы заказчика
- Средний - важное улучшение
- Низкий - второстепенные задачи

Дополнительно учитывалась модель Impact / Effort (влияние / трудозатраты).
